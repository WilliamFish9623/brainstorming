# 创意评价框架

## 一、评价原则

1. 八个维度必须分开评价，不用单一印象替代逐项判断。
2. 创意数量不等同于创意质量。候选项多，只表示搜索范围较大，不代表其中必然存在高质量方案。
3. 禁止无依据的伪精确。证据不足时，不给出看似精确的百分比、收益额、工期或小数评分。
4. 每项判断都应附上依据、证据等级和主要不确定性；无法判断时明确写“信息不足”。
5. 先评价，后排序。不得为了迎合预设结论而倒推各维度评价。
6. 高、中、低是定性锚点，不是精密测量值。相邻等级难以区分时，应保留并列或标记“中高”“中低”，而不是强行拉开差距。
7. 评价应基于同一目标、同一约束和相近的时间范围。评价基准变化时，应重新评价。

## 二、八个评价维度

### 1. 新颖性

定义：相对于目标用户、现有方案、组织既有实践和可获得替代方案，该创意提供新机制、新组合或新体验的程度。

高：核心机制或组合方式明显不同于常见方案，且这种差异与问题相关，不只是表面包装。

中：沿用已知机制，但在场景、流程、组合或体验上有有意义的改进。

低：与现有方案高度相似，差异主要是命名、样式或轻微调整。

注意：新颖不自动代表有价值，也不自动代表不可行。

### 2. 目标匹配

定义：创意对既定问题、目标用户、使用场景、成功标准和约束条件的贴合程度。

高：直接解决核心问题，覆盖主要用户与关键场景，并符合明确约束。

中：能解决部分核心问题，但存在场景缺口、用户覆盖不足或次要目标偏离。

低：主要解决边缘问题、替代问题或未经确认的问题，与成功标准关联弱。

### 3. 可行性

定义：在现有技术、组织能力、法规条件、依赖关系和实施环境下实现该创意的可能性。

高：关键能力已具备，依赖清晰，有可参考的实现路径，主要障碍可控。

中：总体可实现，但存在需要验证的技术、协作、合规或供应依赖。

低：依赖尚不存在的能力、重大外部条件或未解决的关键障碍。

### 4. 预期价值

定义：创意成功后对用户、业务、组织或社会目标可能产生的有效改善，包括收益范围、改善幅度和持续性。

高：对核心指标或关键体验有显著且可持续的正向影响，受益对象明确。

中：可产生有意义的改善，但影响范围、幅度或持续性有限。

低：改善轻微、受益对象模糊，或价值主要依赖未经验证的假设。

注意：在没有数据支撑时使用方向性描述，不编造收益数字。

### 5. 风险

定义：实施或运行过程中造成损失、伤害、合规问题、声誉影响、机会成本或不可逆后果的可能性与严重程度。

高：可能造成严重或难以逆转的后果，且缺少成熟缓解措施。

中：存在实质性负面后果，但可通过设计、监控、试点或流程控制降低。

低：负面后果范围有限、可逆，且已有明确缓解措施。

注意：风险等级越高越不利；应同时记录风险类型、触发条件和缓解办法。

### 6. 资源时间

定义：从验证到实施、上线和维护所需的人力、资金、数据、设备、协作成本与日历时间。

高：需要大量或稀缺资源，周期长，跨团队依赖多，持续维护负担重。

中：需要可获得但不轻量的资源，周期和协作成本可管理。

低：可由现有团队在较短周期内完成，依赖少，维护负担有限。

注意：资源时间等级越高越不利；缺少估算基础时使用区间或相对描述。

### 7. 验证成本

定义：在获得足以支持继续、调整或停止决策的证据前，需要投入的资源、时间和接触真实环境的难度。

高：必须建设接近完整的方案、获取稀缺数据或经历长周期后才能验证关键假设。

中：需要原型、有限开发、用户测试或小规模试点才能获得有效证据。

低：可通过访谈、纸面原型、模拟、已有数据或小实验快速验证关键假设。

注意：验证成本等级越高越不利，但高价值创意不应仅因验证成本高而直接淘汰。

### 8. 不确定性

定义：关于问题、用户需求、机制有效性、实施条件、价值和风险等关键判断中未知因素的数量与重要程度。

高：多个关键假设未经验证，结果对外部条件敏感，现有证据弱或相互矛盾。

中：主要方向有一定依据，但仍有少数关键假设会显著影响结论。

低：关键假设已有直接证据支持，边界条件明确，结果相对稳定。

注意：不确定性等级越高越不利；不确定性不是风险本身，而是对结果和风险缺乏把握。

## 三、证据等级

每个维度的评价都应标注以下证据等级之一：

| 等级 | 含义 | 典型依据 |
|---|---|---|
| E4 强 | 有直接、重复或真实环境证据 | 已运行数据、对照实验、多个一致试点、真实行为数据 |
| E3 较强 | 有直接但范围有限的证据 | 单次试点、原型测试、目标用户行为观察、可靠内部数据 |
| E2 一般 | 有间接证据或结构化专家判断 | 相邻场景案例、行业资料、系统性访谈、专家评审 |
| E1 弱 | 主要依赖假设、类比或少量意见 | 个别反馈、未经验证的推断、非目标用户意见 |
| E0 无 | 当前没有可用依据 | 待验证假设、信息缺失 |

证据等级描述的是判断依据的强弱，不直接决定创意优劣。高评价配低证据时，应优先验证，而不是直接视为确定优势。

## 四、相对排序法

当精确数据不足时，优先使用相对排序，而不是编造综合分数。

1. 明确本轮必须满足的目标与硬约束。
2. 对每个创意逐维给出高、中、低评价，并标注证据等级。
3. 在同一维度内进行候选项比较，优先判断明显优于、近似或明显劣于。
4. 先识别支配关系：若方案甲在关键正向维度不差于方案乙，在至少一个关键维度更好，且负向维度不更差，则甲可排在乙之前。
5. 无支配关系时，按目标优先级比较差异最大的关键维度。
6. 难以区分时允许并列，并列项通过低成本验证获得新证据后再排序。
7. 排序结果应附理由，不把高、中、低机械转换成伪精确总分。

正向维度为新颖性、目标匹配、可行性、预期价值，通常等级越高越有利。负向维度为风险、资源时间、验证成本、不确定性，通常等级越低越有利。

## 五、可选用户权重法

仅在用户能明确表达优先级时使用权重。权重用于体现偏好，不用于制造客观精确的假象。

1. 请用户把有限的权重点分配给八个维度，例如总计 100 点；也可只标注“关键、重要、次要”。
2. 权重必须由用户目标和约束导出，不由评价者代替用户臆测。
3. 若需要辅助排序，可将高、中、低临时映射为有序值，但结果只作为排序提示，不作为精确效用值。
4. 对负向维度反向处理，确保高风险、高资源时间、高验证成本和高不确定性不会被误当成优势。
5. 同时展示原始八维评价、权重和排序理由，不只展示一个综合结果。
6. 对低证据评价进行显著标记。必要时分别给出“按当前判断的排序”和“按证据可信度调整后的排序”。
7. 当轻微权重变化会改变名次时，标记为“排序敏感”，不宣称存在稳定最优方案。

## 六、排序流程

1. 确认目标：写明要解决的问题、目标用户、时间范围、成功标准和硬约束。
2. 清理候选：合并实质重复项，但保留差异说明；候选数量不进入质量评价。
3. 独立评价：逐个创意、逐个维度判断，避免先看总印象或预期名次。
4. 标注依据：为每项评价写一句依据，并标注 E0 至 E4。
5. 检查一致性：确认同一等级在不同候选项中使用了相同标准。
6. 处理门槛：违反硬约束或存在不可接受风险的方案单独列为“不进入当前排序”，而不是用其他优势抵消。
7. 相对比较：在各维度内比较候选项，识别支配关系、明显差异和并列项。
8. 应用偏好：默认依据已确认的目标优先级；如用户提供权重，再进行权重辅助排序。
9. 检查证据：识别高评价低证据、低评价低证据及证据冲突项。
10. 形成结论：给出优先级、主要理由、关键权衡、待验证假设和下一步最小验证动作。
11. 复评：获得新证据或目标变化后，更新相关维度和排序，不沿用失效结论。

## 七、冲突处理

### 维度之间冲突

1. 目标匹配优先于表面新颖。新颖性高但目标匹配低的方案通常不应优先。
2. 高价值与高风险并存时，不直接抵消；先判断风险是否触及不可接受门槛，再讨论缓解措施。
3. 高价值与高资源时间并存时，比较分阶段实施、缩小范围或先验证关键机制的可能性。
4. 可行性低但验证成本低时，可先做验证，不必立即淘汰。
5. 不确定性高但验证成本低时，优先安排实验以减少未知。
6. 证据弱时，不用强烈措辞得出确定结论。

### 评价者之间冲突

1. 先检查双方使用的目标、约束、时间范围和维度定义是否一致。
2. 要求各方分别陈述依据和证据等级，区分事实分歧、假设分歧与价值偏好分歧。
3. 对事实分歧，寻找更直接证据或设计验证。
4. 对假设分歧，记录各自成立条件，并进行情景比较。
5. 对价值偏好分歧，由决策者或用户明确权重与不可接受边界。
6. 证据仍不足时保留并列或条件式结论，不以平均意见伪装共识。

### 排名冲突

1. 检查是否存在硬约束或一票否决风险。
2. 检查排名是否对权重变化敏感。
3. 检查关键评价是否仅有 E0 或 E1 证据。
4. 若无法稳定区分，输出候选集而不是虚构唯一第一名。
5. 优先验证最可能改变排序的关键假设。
