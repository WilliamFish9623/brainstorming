# 头脑风暴 Skill 设计依据

## 1. 文档目的

本文说明 `brainstorming` Skill 的设计依据、证据边界、仓库适配方式和预期工作流。目标不是把 Azzedde/brainstormers 原样改写成中文，而是在保留其六种方法、提示模板分层和树形追踪思想的基础上，补足面向中文实际任务所需的澄清、约束管理、证据标记、去重、风险门槛、收敛与行动闭环。

本文不声称六种方法都具有同等强度的学术证据，也不把第三方页面的命名、解释或商业模板视为方法官方定义。

## 2. 分析范围与证据状态

### 2.1 分析范围

本次分析覆盖：

- Azzedde/brainstormers 仓库 `main` 分支提交 `bd43fdfcaf9ea23afc99ec5183a1b4077f599cdc` 的可见实现与 README。
- `src/lib/brainstorm/methods.ts` 的六方法元数据。
- `src/lib/brainstorm/prompts.ts` 的初始生成与扩展模板。
- `src/types/brainstorm.ts` 的方法、提示、消息、会话和树节点类型。
- 仓库对列表输出的格式约定和树形结果追踪方式。
- 与方法命名或方法边界相关的外部资料，重点核对 Big Mind Mapping、Role Storming 和 Six Thinking Hats 的来源表述。
- 当前工作区中 Skill 的目标目录结构与验收需求。

不在本次范围内：

- 对六种方法做受控效果实验。
- 证明某一方法能提高创意数量、质量或业务结果。
- 对第三方方法商标、授权或法律状态作独立法律判断。
- 修改上游 Next.js 应用或运行其在线服务。
- 为高风险领域提供替代专业审查的结论。

### 2.2 证据状态分级

| 状态 | 含义 | 本文使用方式 |
|---|---|---|
| 已核对仓库事实 | 可由指定提交中的源码、类型或 README 直接确认 | 可作为兼容设计依据，但不自动等于最佳实践 |
| 已核对外部证据 | 可由列明的外部页面或论文直接确认 | 只支持其明确陈述，不外推到未研究场景 |
| 未证实或未找到 | 指定来源未出现目标说法，或当前材料不足 | 明确写为缺口，不用其他相似说法补齐 |
| 设计推断 | 为满足中文 Skill 使用目标而提出的工程选择 | 必须与事实分开，不包装成仓库原有能力或研究结论 |
| 待实测 | 需要在目标模型、真实任务或自动化验收中验证 | 形成测试项，不先写成确定效果 |

### 2.3 当前证据结论

- 已核对仓库事实：指定提交是 Next.js 应用，使用 TypeScript、React、Tailwind 等前端技术；不是 Skill 仓库。
- 已核对仓库事实：根目录可见结构没有 Skill、`references`、`tests` 规范；页面也未展示自动测试。严谨表述应是“在指定提交的可见结构和说明中未见”，而不是证明任何历史版本绝对不存在。
- 已核对仓库事实：实现提供六个固定方法 ID，并为每种方法配置元数据、初始提示和扩展提示。
- 已核对仓库事实：提示要求项目符号以 `- ` 开头；多数初始模板还要求不要编号、标题或加粗文本。
- 已核对仓库事实：`TreeNode` 保存 `methodUsed` 等追踪字段，可表达节点层级、父子关系和生成方法。
- 已核对外部证据：论文真实标题为 `Eliciting Topic Hierarchies from Large Language Models`，研究主题是用 LLM 构造不同具体程度的话题层级；论文标题不是 Big Mind Mapping。
- 未证实或未找到：指定 Psychology Today 原文中未见 Role Storming，因此不能把该原文列为 Role Storming 的直接来源依据。
- 已核对外部证据：GroupMap 当前页面标题为 `Six Thinking Styles`，页面同时说明其不是 Edward de Bono Ltd 的附属、认可或官方授权产品。
- 待实测：中文 Skill 增加的约束门槛、组合方法、去重和行动闭环是否提升输出质量，需要通过验收案例与真实任务验证。

## 3. 仓库适配

### 3.1 上游仓库定位

Azzedde/brainstormers 在指定提交中是一个交互式 Web 应用。它通过方法选择、提示模板、模型响应解析和树结构展示来组织头脑风暴。Skill 目标则是让模型在对话或 CLI 环境中完成需求澄清、方法选择、结构化发散、审查、收敛和行动化。

因此采用“概念与协议适配”，不采用“应用代码搬运”：

- 保留六个方法 ID，便于追踪和对照。
- 保留 initial/expansion 两阶段思想。
- 保留 `- ` 项目符号兼容要求。
- 借鉴 `TreeNode.methodUsed` 的方法追踪思想。
- 不复制 UI、状态管理、API key 管理、流式组件或 Next.js 路由。
- 不声称 Skill 目录结构来自上游规范，因为上游没有展示该规范。

### 3.2 六个方法 ID

固定方法 ID 如下：

- `big-mind-mapping`
- `reverse-brainstorming`
- `role-storming`
- `scamper`
- `six-thinking-hats`
- `starbursting`

这些 ID 用于方法选择、中间产物追踪、验收和未来结构化导出，不翻译为中文标识符。

### 3.3 methods.ts 元数据适配

仓库中每个方法元数据包含：

- `id`
- `name`
- `description`
- `icon`
- `gradient`
- `textGradient`
- `whenToUse`
- `examples`

Skill 适配规则：

- 保留 `id`、`name`、`description`、`whenToUse`、`examples` 的语义用途。
- `icon`、`gradient`、`textGradient` 属于界面表现字段，纯文本 Skill 不依赖它们。
- 中文说明不能改变固定 ID。
- `whenToUse` 只作为方法选择提示，不作为排他性真理；实际选择还需看目标、阶段、约束与风险。

### 3.4 prompts.ts 模板适配

仓库为每个方法提供 `initial` 和 `expansion` 两个模板。

已核对变量：

| 方法 | initial 变量 | expansion 变量 |
|---|---|---|
| `big-mind-mapping` | `{userInput}` | `{idea}` |
| `reverse-brainstorming` | `{userInput}` | `{idea}` |
| `role-storming` | `{userInput}` | `{role}`、`{originalTopic}`、`{idea}` |
| `scamper` | `{userInput}` | `{technique}`、`{originalTopic}`、`{idea}` |
| `six-thinking-hats` | `{userInput}` | `{hatType}`、`{originalTopic}`、`{idea}` |
| `starbursting` | `{userInput}` | `{originalTopic}`、`{category}`、`{idea}` |

适配规则：

- Skill 继续区分初始发散和选中项扩展，但不机械执行固定数量。
- 数量服从用户时间、上下文、重复率和风险要求；偏离仓库默认数量时应说明是轻量或深度适配。
- 扩展前必须先选中值得深化的项，避免所有分支无差别膨胀。
- 模板变量只能填入已有输入、明确假设或已生成节点，不补造外部事实。
- 方法切换时保留目标、硬约束、已淘汰项和待确认问题，不能只保留表面主题。

### 3.5 输出解析适配

仓库提示以 `- ` 作为项目符号，响应解析通常依赖一致格式。Skill 中需要机器兼容的创意列表时：

- 每个可解析创意项以 `- ` 开头。
- 不在同一解析区混用 `1.`、`*` 或无前缀行。
- 标签应置于项目符号之后，例如 `- WHO:` 或 `- SUBSTITUTE:`。
- 解释性章节、表格和行动清单可采用 Markdown 结构，但若要回灌仓库解析器，应先转换为纯 `- ` 列表。
- 不使用 Markdown 加粗作为语义载体。

### 3.6 TreeNode 追踪适配

仓库 `TreeNode` 字段为：

- `id`
- `content`
- `level`
- `parentId?`
- `children`
- `methodUsed`
- `timestamp`

Skill 不必强制输出 JSON 树，但中间产物应保留等价追踪信息：

- 每个想法知道来源方法。
- 扩展项能回连父想法。
- 方法切换后记录新方法，不覆盖原来源。
- 合并、淘汰和保留理由可追踪。
- 事实、假设与创意不能在树中无标记混合。

### 3.7 测试现状与补足

指定提交中未见自动测试和 Skill 验收规范。当前设计新增 `brainstorming/tests/acceptance-cases.md` 作为人工或模型评测用例，至少覆盖：

- 信息不全
- 目标模糊
- 强约束
- 高风险
- 多角色
- 重复率高
- 方法适配与跳过
- 输出解析与行动闭环

这是一项设计补足，不是上游仓库已有测试能力。

## 4. 事实、外部证据与设计推断三分

### 4.1 仓库事实

以下判断仅描述指定提交：

- 项目为 Next.js 应用，而非 Skill 包。
- 有六种固定头脑风暴方法。
- 方法元数据字段为 `id/name/description/icon/gradient/textGradient/whenToUse/examples`。
- 提示层分为 `initial` 与 `expansion`。
- 初始与扩展提示使用占位变量。
- 列表格式要求以 `- ` 开头。
- `TreeNode` 追踪层级、父节点、子节点、`methodUsed` 和时间。
- 可见材料未展示自动测试。

这些事实能支持“如何兼容”，不能证明方法有效性或输出正确性。

### 4.2 外部证据

- `Eliciting Topic Hierarchies from Large Language Models` 讨论 LLM 辅助话题范围界定和层级生成。它可以支持“层级化话题探索需要控制具体程度、相关性和重复”的设计关注点。
- 该论文不能直接证明上游命名的 Big Mind Mapping 是论文方法，也不能证明任意递归扩展都有效。
- 指定 Psychology Today 原文未见 Role Storming，因此不得写成 Role Storming 的直接来源或认证。
- GroupMap 当前页面使用 `Six Thinking Styles`，并对 Six Thinking Hats® 作识别性说明，同时明确非附属、非认可、非官方授权。
- 外部页面可能变化；引用时应保留访问边界，不将商业页面当作唯一权威定义。

### 4.3 设计推断

以下内容是本 Skill 为提高实用性而新增的工程规则：

- 先澄清目标和约束，再选方法。
- 在信息不足但必须继续时显式列假设。
- 高风险任务先做门槛与停止条件，不先发散执行细节。
- 多方法可以串联，但每个方法必须有明确职责。
- 数量不是质量，重复率高时允许少于目标数量。
- 发散后必须去重、筛选、排序并形成行动。
- 角色视角是模拟，不得冒充访谈。
- 事实、外部证据、假设、创意和建议分别标记。
- 精确数字必须来自输入、可复现计算或可靠来源；否则标记待确认。

这些规则需要通过验收和实际使用继续修订。

## 5. 六方法矩阵

| 方法 ID | 核心动作 | 适合阶段 | 典型输入 | 关键中间产物 | 主要风险 | 跳过条件 | 推荐后续 |
|---|---|---|---|---|---|---|---|
| `big-mind-mapping` | 从主题向多个分支扩展，再逐层具体化 | 早期探索、内容选题、方向不足 | 一个较宽主题和基本边界 | 主题分支、父子节点、具体程度层级 | 分支爆炸、重复、越扩越偏 | 用户已有明确方案且只需改进；时间极短且方向已知 | 去重、门槛筛选、选中少量分支再扩展 |
| `reverse-brainstorming` | 先想如何造成或恶化问题，再反转为防护与机会 | 风险发现、流程改进、失败预防 | 明确目标或问题 | 失败机制、原因、防护措施 | 负面项被误当行动建议；高风险场景产生不当细节 | 问题尚未定义；反向思考会引入不必要危险且无法安全抽象 | 风险分级、责任分配、验证控制措施 |
| `role-storming` | 从不同角色、用户或利益相关者视角生成见解 | 多角色需求、体验设计、冲突识别 | 主题、角色、各方已知约束 | 视角卡、诉求、担忧、冲突 | 刻板印象、替真实群体发言、虚构调研 | 已有可靠访谈数据且任务仅需归纳；角色与问题无关 | 用真实利益相关者核验，形成冲突与折中表 |
| `scamper` | 对现有方案做替代、组合、调整、修改、作他用、删减、反转 | 现有产品、服务或流程改进 | 一个已有对象及其问题 | 七类变换、候选改造、代价 | 机械套七项、只改表面、不看副作用 | 没有可改造对象；目标是完全开放探索 | 比较新代价，做小规模验证 |
| `six-thinking-hats` | 分离事实、情绪、风险、收益、创意和过程视角 | 综合审查、复杂决策、团队对齐 | 候选方案、事实与约束 | 六视角记录、信息缺口、审查结论 | 把帽子内容当证据；六块平均用力 | 任务只需快速发散；候选和事实都不存在 | 蓝帽收敛为决策、待办和复查点 |
| `starbursting` | 用 WHO、WHAT、WHERE、WHEN、WHY、HOW 系统提问 | 需求发现、项目规划、目标澄清 | 一个主题或拟议项目 | 5W1H 问题、未知项、责任边界 | 只提问不回答；问题过多不收敛 | 用户已提供完整需求且只需改造或选择 | 按影响排序问题，补证据后进入方案生成 |

### 5.1 方法组合原则

- 默认只选一个主方法；只有单一方法无法覆盖任务时才组合。
- 每增加一个方法，都要说明它解决哪个未覆盖问题。
- 常用组合一：`starbursting` 定义问题，`big-mind-mapping` 发散，筛选后行动化。
- 常用组合二：`role-storming` 识别多方诉求，`six-thinking-hats` 综合审查。
- 常用组合三：`reverse-brainstorming` 找失败机制，`scamper` 改造现有方案。
- 高风险任务中，方法组合不能替代专业核验与审批。
- 方法切换不是重新开始；必须继承目标、约束、证据状态、已淘汰项和待确认问题。

## 6. 总工作流

### 步骤 0：判断任务是否适合头脑风暴

输入：用户请求、任务类型、所需结果。

处理：判断用户需要的是创意生成、问题探索、方案改造还是决策审查。如果答案可由确定事实、简单计算或直接执行得到，则不强行套用头脑风暴。

输出：进入头脑风暴，或改用直接回答、检索、计算、执行等方式。

跳过条件：用户明确要求使用某方法时可直接进入方法准备，但仍要检查安全和硬约束。

### 步骤 1：提取目标与交付物

输入：用户原始表达。

处理：提取想解决的问题、目标受众、期望交付物、使用场景和成功判据。将“更好”“创新”“有影响力”等模糊词改写为可选择或可检查的目标。

输出：一句目标陈述、交付物清单、成功判据或待选目标。

跳过条件：目标、交付物和成功判据已经明确且相互一致。

### 步骤 2：盘点信息与证据状态

输入：用户提供的事实、文件内容、外部资料、历史结论。

处理：分为已知事实、外部证据、待确认信息、工作假设、设计偏好。对数字、来源和因果表述做边界检查。

输出：证据状态表和关键缺口。

跳过条件：不可完全跳过；简单低风险任务可压缩为一句假设声明。

### 步骤 3：识别约束、风险与利益相关者

输入：预算、时间、资源、禁止项、法律伦理边界、角色诉求。

处理：区分硬约束与软偏好，检查约束冲突；高风险任务建立启动门槛和停止条件；多角色任务建立利益相关者地图。

输出：约束清单、冲突项、风险门槛、角色表。

跳过条件：低风险个人创意任务且没有明显约束或其他受影响方时，可只记录时间与输出限制。

### 步骤 4：决定澄清还是带假设继续

输入：关键缺口、用户时间要求、任务风险。

处理：如果缺口会改变安全性、合法性或方案方向，则先提问；如果任务低风险且用户时间有限，则列出最小工作假设继续。

输出：高价值澄清问题，或明确标记的假设集。

跳过条件：信息已足够，或用户已经授权在明确假设下继续。

### 步骤 5：选择主方法和深度

输入：任务阶段、目标、已有方案、约束、时间、风险和角色复杂度。

处理：依据六方法矩阵选择一个主方法；必要时指定一个辅助方法。决定轻量、标准或深度模式。

输出：方法 ID、选择理由、预期轮次、跳过的方法及原因。

跳过条件：用户指定方法且适配任务；若明显不适配，应简短指出并征得是否调整。

深度建议：

- 轻量：一次发散、一次筛选、直接行动，适合时间有限与低风险任务。
- 标准：澄清、发散、扩展、去重、评价、行动。
- 深度：多方法串联、利益相关者核验、风险门槛、证据缺口、候选比较和阶段决策。

### 步骤 6：生成初始想法或问题

输入：方法、目标、约束、假设和禁止项。

处理：按方法生成差异明显的第一层节点。每项应完整、相关、可理解；需要仓库兼容时以 `- ` 开头。

输出：初始想法、问题、角色视角、失败机制或方法分类项。

跳过条件：用户已有足够候选，只需评价或扩展时跳过宽泛初始生成。

### 步骤 7：去重与覆盖检查

输入：新生成项、用户已有项、历史淘汰项。

处理：按语义目标、作用机制、受众、资源和结果判重，而非只比较字面。检查是否覆盖不同分支，合并近义项并记录理由。

输出：去重后的候选、合并映射、覆盖缺口。

跳过条件：候选很少且差异显然，但仍需进行快速检查。

### 步骤 8：选择性扩展

输入：通过初筛的候选及其父节点。

处理：只扩展高价值或信息不足的节点，使用相应 expansion 变量；保持父子关系和 `methodUsed` 追踪。扩展应探索实现、影响、变化、应用和挑战，而不是同义改写。

输出：二级或三级想法树、扩展理由。

跳过条件：时间极短、初始项已足够具体、用户只要候选标题或继续扩展会造成重复。

### 步骤 9：约束回检与风险审查

输入：候选方案、硬约束、风险门槛、证据状态。

处理：逐项检查可行性、边界、资源、隐私、安全、公平和未经证实的承诺。高风险内容标记需要专业核验的部分。

输出：不合格项、修改项、保留项和待核验事项。

跳过条件：不可完全跳过；低风险任务可简化为一行约束检查。

### 步骤 10：收敛与排序

输入：合格候选、用户成功判据、资源限制。

处理：使用少量、透明的标准做定性或有依据的评分。评分只作会话内相对比较，不虚构市场概率、效果率或精确收益。

输出：优先候选、排序理由、未选项原因和不确定性。

跳过条件：用户明确只要求发散且不要筛选；仍应提供简单分组方便后续使用。

### 步骤 11：形成最终输出

输入：优先候选、证据状态、约束审查结果。

处理：按用户所需格式整理。至少说明方案内容、适用条件、关键假设、风险或限制。虚构案例与模拟视角必须显式标记。

输出：可读、可复用、不过度承诺的方案或创意集合。

跳过条件：不可跳过。

### 步骤 12：行动化与验证

输入：最终候选、可用资源和时间。

处理：把方案转换为下一步行动，包含交付物、责任角色或待确认责任人、截止点、依赖、验证方式和停止条件。优先设计小规模、可逆的验证。

输出：行动清单、验证计划、待确认问题和复查节点。

跳过条件：用户明确只需原始创意；此时至少给出一个最小下一步，不强行展开项目计划。

### 步骤 13：保存追踪信息

输入：本轮方法、节点、筛选与行动。

处理：记录每个节点的方法来源、父节点、状态和关键变更；方法切换时保留上下文。

输出：可供后续继续的会话摘要或结构化树。

跳过条件：一次性极短回答且用户不需要继续时可省略显式展示，但内部组织仍应保持来源清晰。

## 7. 结构概览

目标 Skill 的最小结构为：

```text
brainstorming/
├── SKILL.md
├── references/
│   └── examples.md
└── tests/
    └── acceptance-cases.md
```

各文件职责：

| 文件 | 职责 | 不应承担 |
|---|---|---|
| `SKILL.md` | 触发条件、硬规则、方法选择、工作流、输出协议 | 塞入过长案例或研究综述 |
| `references/examples.md` | 提供轻量与深度完整示例，展示中间产物和行动 | 把虚构案例写成真实案例研究 |
| `tests/acceptance-cases.md` | 提供验收输入、预期、禁止和最低标准 | 声称已自动执行或已获得统计通过率 |
| `中文头脑风暴Skill设计依据.md` | 记录证据、适配理由、边界和设计推断 | 取代 Skill 的执行规则或外部专业意见 |

未来只有在存在真实重复逻辑或自动执行需求时，才考虑新增脚本；不为假设性需求预建工具。

## 9. 通用限制

### 9.1 事实与数据

- 不虚构用户研究、案例效果、成本、人数、日期、通过率或引用。
- 精确数字须来自用户输入、可复现计算或已核对来源。
- 演示所需的虚构数字必须标记为虚构，不得与现实事实混排。
- 角色模拟不是访谈，模型推断不是用户证据，候选评分不是市场预测。

### 9.2 方法有效性

- 六方法是结构化思考工具，不保证产生新颖、正确、可行或有效的结果。
- 递归扩展可能放大错误、偏见、重复和主题漂移。
- 固定数量要求可能诱发凑数；质量不足时允许减少数量并说明原因。
- 多方法串联会增加上下文和重复，必须有明确分工。

### 9.3 高风险任务

- 涉及未成年人、医疗、法律、财务、安全、隐私或重大资源决策时，先标明专业核验、审批和停止条件。
- 头脑风暴不能替代资质审查、风险评估、法律意见或现实利益相关者参与。
- 反向头脑风暴只生成足以识别风险的抽象失败机制，不扩展为可被滥用的实施指南。

### 9.4 中文表达与解析

- 正文以中文为主，固定方法 ID、字段名、文件名和必要术语保留英文。
- 用明确标签控制“必须、应、可、禁止”的约束强度。
- 机器解析列表保持 `- ` 前缀一致。
- 不依赖 Markdown 加粗表达硬约束。
- 避免把长复合句中的限制当作背景信息；关键约束单独列出。

### 9.5 来源与版本

- 仓库结论绑定到提交 `bd43fdfcaf9ea23afc99ec5183a1b4077f599cdc`，后续版本可能变化。
- 外部网页内容和标题可能更新，重要判断应保留访问日期或重新核验。
- “未见”表示在指定材料和分析范围内未找到，不等于对整个互联网或全部历史版本的绝对否定。
- 商标、授权和法律状态以权利方与适用法律的正式信息为准，本文不作法律结论。

## 10. 验证计划

当前没有可引用的自动测试结果。最低验证路径为：

1. 用 `tests/acceptance-cases.md` 的全部案例逐项执行。
2. 检查每例是否满足预期、未触发禁止项并达到最低标准。
3. 对同一输入至少复测一次，观察约束遵循和重复率是否稳定。
4. 对轻量与深度示例检查事实、假设、模拟视角和虚构标记是否清晰。
5. 对需要仓库兼容的输出执行 `- ` 前缀解析检查。
6. 记录失败类别，不用单一“通过率”掩盖高风险失败。
7. 在目标模型或 Skill 规则更新后重新验收。

完成这些验证前，只能说“设计覆盖了相应场景”，不能说“Skill 已被证明有效”。
